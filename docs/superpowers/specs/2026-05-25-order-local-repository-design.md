# 订单本地仓库与全部账号视图设计

## 背景

订单管理需要新增“全部账号”视图，并调整订单数据请求结构。订单页不应在每次切换账号时直接请求微信小店接口，而应从统一的本地订单仓库读取数据。后台负责定期同步各账号订单，单账号和全部账号都可以手动触发刷新。

本设计只处理插件本地业务流程和仓库接口，不处理长期数据库选型。后续数据量变大时，可以把仓库实现从临时存储替换为 IndexedDB 或其它本地数据库，而不改订单页、同步中心和 IPC 业务接口。

本设计也不改变已有“订单云同步”设计。微信 API 可重新获取的订单详情、订单状态和订单列表仍不进入云同步范围。

## 用户体验

订单管理左侧账号栏新增“全部账号”入口。

- 选中“全部账号”时，订单表展示所有账号的本地订单快照。
- 选中单个账号时，订单表只展示该账号的本地订单快照。
- 切换账号或切换到全部账号不直接请求微信接口。
- 工具栏展示上次更新时间、同步状态、自动更新倒计时和“立即更新”按钮。
- 自动同步周期第一版为 1 分钟。
- “立即更新”在单账号视图只刷新当前账号，在全部账号视图刷新所有账号。

搜索支持两种模式：

1. 本地搜索
   - 默认模式。
   - 从本地订单仓库按当前视图范围搜索。
   - 响应快，不触发微信接口。

2. 服务器最新搜索
   - 用户显式选择后触发。
   - 单账号视图只搜索当前账号。
   - 全部账号视图跨所有账号搜索，并合并结果。
   - 搜索结果回写本地订单仓库，后续本地列表和本地搜索可以复用。

## 架构

新增三个后台业务边界。

### OrderRepository

订单仓库接口，隐藏底层存储实现。第一版可使用轻量临时实现，但调用方只能依赖接口。

核心能力：

- `upsertOrders(accountId, orders, source)`：写入或更新账号订单快照。
- `listOrders(scope, filters)`：按全部账号或单账号范围读取订单。
- `searchLocal(scope, params)`：在本地仓库搜索订单。
- `getOrder(accountId, orderId)`：读取单个订单快照。
- `getSyncState(scope)`：读取上次同步时间、同步中状态、下次同步时间和错误摘要。
- `markSyncStarted(scope)` / `markSyncFinished(scope, result)`：记录同步状态。

订单记录必须保留真实 `accountId` 和账号名称快照，供全部账号表格展示以及发货、地址解密、关联采购单等动作定位真实账号。

### OrderSyncCenter

订单同步中心，负责自动同步和手动刷新。

职责：

- 使用现有 `ScheduledJob` 调度中心触发每分钟同步，不在订单业务模块中单独创建 `chrome.alarms`、`setInterval` 或其它散落循环。
- 串行或小并发刷新账号，避免所有账号同时打满微信接口。
- 单账号失败不影响其它账号。
- 去重同一账号的并发刷新请求。
- 同步完成后写入 `OrderRepository`。
- 记录账号级错误摘要，供 UI 展示。

第一版自动同步按账号拉取订单列表首批数据并回写仓库，默认每账号保留最近 `500` 条订单快照。历史深度分页和长期归档不在第一版展开；用户需要确认最新结果时使用“服务器最新搜索”。

### OrderQueryService

订单页和 runtime handler 面向的业务查询服务。

职责：

- 普通列表读取本地仓库。
- 本地搜索读取本地仓库。
- 服务器最新搜索调用微信接口，成功后回写本地仓库。
- 订单详情优先读本地快照；当本地没有该订单或用户动作需要最新详情时，再调用微信接口并回写仓库。

## IPC 边界

新增或调整类型化 runtime channel，仍通过 `src/shared/runtime-channels.ts` 和 `extensionApi` 暴露。

第一版新增：

- `orders:listLocal`
- `orders:searchLocal`
- `orders:searchRemote`
- `orders:refresh`
- `orders:getSyncState`

现有直接请求微信接口的 `orders:list` 和 `orders:search` 可以先保留为兼容入口，但订单页新流程不再直接使用它们。

## UI 调整

`Layout` 的账号侧边栏在订单管理模块显示“全部账号”。订单页接收一个订单范围，而不是只接收单个 `accountId`。

订单表在全部账号视图需要显示账号列。单账号视图可以隐藏账号列。

需要保留现有订单动作能力：

- 查看详情
- 解密地址
- 货源管理
- 采购单关联
- 采购发货状态检测
- 回填发货
- 淘宝退款流程

这些动作必须使用订单记录上的真实 `accountId`，不能使用“全部账号”虚拟 ID。

## 错误处理

- 自动同步失败只展示摘要，不阻断本地已有数据展示。
- 单账号凭证失效时，通过现有 credential error 机制提示用户，并在同步状态中标记该账号失败。
- 全部账号刷新时，部分账号失败仍返回成功账号的数据和失败账号摘要。
- 服务器最新搜索跨账号执行时，部分账号失败不清空已有本地结果。

## 非目标

第一版不做：

- IndexedDB schema 设计。
- 长期订单归档策略。
- 云端保存微信订单详情或订单列表。
- 按字段增量同步订单详情。
- 复杂全文索引。
- 后台持续常驻 `setInterval`。

## 验证

实现后至少验证：

- `npm run compile`
- 订单管理可切换全部账号和单账号。
- 切换视图不触发远程列表请求。
- 每分钟自动同步会更新仓库状态。
- 手动刷新单账号只刷新该账号。
- 手动刷新全部账号能处理部分账号失败。
- 本地搜索不触发远程请求。
- 服务器最新搜索会回写本地仓库。
- 全部账号订单动作使用真实账号 ID。
